# Engineering Document Proofreader

エンジニアリングの報告系ドキュメント (障害調査・RCA・リリース報告・設計メモ等) を、**報告文書として読み手に届く文体**に校正するスキル。

内容の正しさは著者が担保済みの前提で、**文体・構造・曖昧表現**の 3 層を点検する。

## いつ使うか

- 障害報告 / RCA / Post-mortem を書き終えた直後
- 調査結果を他チームに共有する Markdown / Notion 用ドキュメントの仕上げ

### 対象文書の種別判定 (最初に必ず行う)

Step 1 でドキュメントを読んだら、次のどちらに該当するかをまず判定する:

- **報告系**: 障害報告 / RCA / Post-mortem / リリース報告 / 影響調査 → 原則 1 を**フル適用**する
- **議論系**: RFC / 設計提案 / トレードオフ分析 / 探索的調査メモ → 原則 1 を**弱める** (仮説・選択肢を示す hedge は保持、構造面のみ指摘)

議論系と判定した場合は、**ステップ 3 のレポート生成前に** ユーザーに次を確認する:

```
このドキュメントは RFC / 設計提案と判定しました。原則 1 (hedge 削除・断言化) は議論を固めすぎるリスクがあるため、以下のどれで進めますか:

1. (推奨) 原則 2 (構造・文章技法) のみ適用する
2. 原則 1 も部分適用する (hedge ごとに根拠有無を個別確認)
3. 実は報告文書として扱いたい (原則 1 フル適用)
```

確認依頼と同時に、1. の方針で出した暫定レポート (原則 2 + 構造面のみ) を併記してよい。ユーザーが即判断できるようにするため。

## 校正の 3 原則

### 原則 1: 断言する (Jun Ishizaka フィードバック由来)

報告文書は基本的に**断言する**。間違っていれば訂正すればよい。

以下の「含みを残す結び」を検出し、書き換えを提案する:

| 検出パターン | 問題 | 方針ラベル | 書き換え方針 |
|---|---|---|---|
| 〜と見られる / 〜とみられる | 観測したのか推測したのか不明瞭 | 断定化 or 未検証明示 | 観測なら「〜である」、推測なら根拠付きで「〜から、〜と判断した」 |
| 〜の可能性が高い / 〜可能性がある | 結論を避けている印象 | 断定化 or 未検証明示 | データがあるなら断言、無いなら「〜は未検証」と明記 |
| 〜が推奨される / 〜することを推奨 | 誰が推奨したか不明、責任の所在が曖昧 | 行動主語化 | 「〜する。担当: X / 期限: Y」と主体と行動で書く |
| 〜した可能性 / 影響が及んだとみられる | 事実か仮説か不明 | 断定化 or 未検証明示 | アラート発火・ログ・数値などの**根拠を並べて断定**するか、「未検証」と明記 |
| 〜と思われる / 〜だろう / 〜かもしれない | 断定回避の典型 | 断定化 or 削除 | 削除するか、根拠を追加して断定に変える |
| 〜が望ましい | 行動指示になっていない | 行動主語化 | 「〜する」「担当者: X」と具体化 |

**3 つの方針ラベル** (書き換えレポートで必ずこれを明示する):

- **断定化**: 本文内に一次情報 (ログ・アラート・数値・PR 等) があり、その主張を支えられる → 「〜である」と言い切り、根拠を括弧内や次文で並記する
- **未検証明示**: 一次情報が本文に無い / まだ確認していない → 「〜は未検証」「今後〇〇で確認する」と書き、推測であったことが読者に伝わる形にする
- **行動主語化**: 主張自体は妥当だが、誰がいつ何をするかが曖昧 → 「〜する。担当: X / 期限: Y」の形式に揃える

**判定の指針**:
- データ・ログ・アラート履歴など**一次情報がある** → 断定化
- 一次情報が**無い / 未検証** → 未検証明示
- 主張は妥当だが行動・責任者が曖昧 → 行動主語化
- いずれでも救えない主張 → **主張そのものを削る**

### 原則 2: 論理構造と文章技法 (Yusuke Shimizu skill 由来)

1. **論理構造**: 総論 → 各論 / 原因 → 結果 / 問題 → 解決 の順で並ぶか確認する
2. **一文一義**: 200 字超・読点 4 つ超の文は分割する
3. **主述のねじれ**: 主語と述語が対応しているか確認する
4. **指示語の明確化**: 「これ」「それ」「当該」が何を指すか前文で明示されているか
5. **用語一貫性**: 同一概念に複数表記が混在していないか (例: `Pod` と `ポッド`、`subscription` と `サブスクリプション`)
6. **動詞を核に**: 「〜化」「〜性」「〜の実施」のような名詞化を動詞に戻す
7. **接続詞で論理を明示**: だから / しかし / 一方 / 具体的には など
8. **数字・単位の統一**: 時刻 (JST/UTC)、数値表記、単位を揃える
9. **事実と推測の区別**: 「〜である」(事実) と「〜と推測される」(推測) を文レベルで分ける
10. **段落 = 主題文 + 展開**: 話題が変わったら改行する
11. **図表のキャプション**: 表・mermaid に「何を示す表か」を明記する
12. **不要語削除**: 「基本的に」「一応」「〜について」「〜という」「〜ということ」を削除
13. **引用の完全性**: Slack リンクは投稿者・日時・チャネルを併記する
14. **読者の時間を盗まない**: 読み手が「結局どうなったの」と聞き返す構造を避ける

### 原則 3: 報告文書の型

報告系ドキュメントは以下の要素を**この順で**持つのが望ましい:

1. **結論** (TL;DR) — 何が起きたか・何をしたか・今どうなっているかを 3 文以内で断言
2. **タイムライン** — 日時は JST/UTC を統一、一次ソースへのリンク併記
3. **根本原因** — コード・設定・運用のどこが原因かを単一の断定文で書き、証拠 (PR・ログ) を続ける
4. **影響範囲** — システム単位で列挙。一次情報と推測は別セクションか、文末で明示的に区別
5. **対応** — 実施済み / 未実施 / 担当者 / 期限 を表形式で
6. **再発防止** — 行動主語で書く (「〜する」「〜を監視する」)

欠けているセクションがあれば指摘する。

## ワークフロー

### Step 1: 読み込みと構造把握

1. 対象ファイルを `Read` で読み込む
2. 見出し構造を抽出し、上記「報告文書の型」と比較する
3. ドキュメント種別を判定する (障害報告 / 調査メモ / リリース報告 / RFC 等)
   - RFC や設計提案の場合は原則 1 を弱めて適用する旨をユーザーに確認する

### Step 2: 検出フェーズ (読み取り専用で完走させる)

次の観点で**全文をスキャン**し、指摘候補を集める。編集はまだしない。

- [ ] 原則 1 のパターン (見られる / 可能性 / 推奨 / 思われる 等) を全箇所抽出
- [ ] 200 字超の文 / 読点 4 つ超の文を抽出
- [ ] 主述のねじれが疑わしい文を抽出
- [ ] 用語の表記揺れ (英日混在・カタカナ / 英字) を抽出
- [ ] 指示語 (「当該」「これ」「それ」) の参照先が曖昧な箇所を抽出
- [ ] 数字・単位・時刻表記の不統一を抽出
- [ ] 不要語 (基本的に / 一応 / 〜について) を抽出

### Step 3: 指摘レポート

検出結果を以下のフォーマットでユーザーに提示する。**最初に一括で出し、ユーザーに採否を判断させる**。

分類は以下の **4 区分を固定** で使う (該当無しの区分も見出しを残し「なし」と書く):

1. **Critical** = 原則 1 の含み表現 (方針ラベル: 断定化 / 未検証明示 / 行動主語化 のいずれかを明記)
2. **Major** = 原則 2 のうち意味に関わるもの (主述ねじれ・論理構造・事実と推測の混在・長文)
3. **Minor** = 原則 2 のうち表記レベル (用語揺れ・不要語・大小文字・数字/単位)
4. **構造** = 原則 3 で欠けているセクション・並び順の指摘

```markdown
## 校正指摘レポート

対象: <path>
種別判定: 報告系 / 議論系 (RFC 判定時はこの位置で確認依頼を挟む)
方針: 以下の指摘を一括提示する。採否判断後、カテゴリ単位で Edit 適用する。

### Critical (原則 1: 含み表現)
- L42 「〜の可能性が高い」 [断定化]
  - 根拠: 〇〇アラートと△△ログが本文 L20 にある
  - 書き換え: 「〜が原因である」
- L70 「影響が及んだとみられる」 [断定化]
  - 根拠: Datadog gRPC エラーの trigger→recover 履歴が本文にある
  - 書き換え: 「短時間の影響が発生した」
- L75 「確認が望ましい」 [行動主語化]
  - 根拠: 確認主体・期限が本文に無い
  - 書き換え: 「〇月〇日までに Datadog で確認する。担当: 未定」

### Major (原則 2: 構造・論理)
- L21 主述のねじれ: 主語「根本原因」に対し述語「実装不具合」(名詞止め)
  - 書き換え: 「〜実装に不具合があった」

### Minor (原則 2: 表記)
- 「subscription」と「サブスクリプション」が混在 → 英字表記に統一
- L17 「基本的に」を削除

### 構造 (原則 3)
- 「対応」セクションが無い。Hotfix 実施者・日時はあるが、未実施アクション (prod アラート再発監視等) の担当者・期限が無い
- 「再発防止」セクションが無い
```

### 末尾の承認依頼

レポートの末尾に、次の形式でユーザーへの質問リストを必ず添える:

```markdown
### 次のアクション (承認待ち)

1. Critical の断定化候補を適用してよいか (該当行の箇条書き)
2. 未検証明示 / 行動主語化の候補は、担当者・期限を埋められるか
3. Major / Minor の適用可否
4. 構造で指摘したセクションを新設してよいか

承認が得られたものから、カテゴリ単位 (Critical → Major → Minor → 構造) で Edit 適用します。
```

### Step 4: 書き換え適用

ユーザーが承認した指摘について、`Edit` で順次書き換える。
- 意味を変える書き換え (推測 → 断定) は**ユーザーに根拠の有無を確認してから**実施する
- 推測を保持する場合は「未検証」「〜の仮説」など、**推測であることを明示**する表現に置換する
- 一度に大量の Edit を連続実行するのではなく、カテゴリ単位 (Critical / Major / Minor) でまとめて差分を出す

### Step 5: 最終チェック

書き換え後に以下を再確認する:

- 原則 1 のパターンが残っていないか grep で確認する
- 結論セクション (TL;DR) を声に出して読んで、「で、結局どうなったの?」と聞き返したくならないか
- mermaid / 表があれば、`mmdc` で変換が通るか確認する (オプション)
